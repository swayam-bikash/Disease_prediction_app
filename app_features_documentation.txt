FEATURE 1 — USER AUTHENTICATION (Login / Signup / Logout)
================================================================================

ROUTES:
  GET/POST  /login
  GET/POST  /signup
  GET       /logout

EXECUTION FLOW:
  Signup:
    1. User fills email + password + confirm password.
    2. Client-side JS validates passwords match before submit.
    3. POST /signup → Flask checks if email already exists in users table.
    4. If duplicate → re-render signup.html with error "Account already exists".
    5. If new → password is hashed with werkzeug generate_password_hash().
    6. Hashed password + email inserted into users table.
    7. Redirect to /login.

  Login:
    1. User submits email + password.
    2. POST /login → Flask queries users table for the email.
    3. check_password_hash() verifies the submitted password against stored hash.
    4. On success → session["user"] = email, session.permanent = True (1-day TTL).
    5. Redirect to /intro (intro video page).
    6. On failure → re-render login.html with error "Invalid email or password".

  Logout:
    1. GET /logout → session.pop("user") clears the session.
    2. Redirect to /login.

  Password Visibility Toggle:
    - JS togglePassword() switches input type between "password" and "text".

  Session Protection:
    - All protected routes check "user" in session; redirect to /login if absent.
    - Admin routes additionally check session["user"] in ADMIN_EMAILS set.

DATABASE:
  Table: users (id, email, password, created_at)

FEATURE 4 — SYMPTOM SELECTION PAGE
================================================================================

ROUTE:
  GET /symptoms → templates/symptoms.html

EXECUTION FLOW:
  1. Flask loads all_symptoms list from Training.csv column names (excluding
     "prognosis" and "Unnamed" columns).
  2. Renders symptoms.html passing the full symptom list.
  3. Each symptom rendered as a checkbox label with data-symptom attribute.

FRONTEND FEATURES:

  4a. Fuzzy Search Autocomplete:
    - User types in search box → JS levenshtein() distance function runs.
    - For queries >= 4 chars: matches symptoms where Levenshtein distance <= 2
      or substring match.
    - For shorter queries: substring match only.
    - Dropdown shows up to 8 results; clicking a result checks that symptom's
      checkbox and scrolls it into view.
    - Keyboard navigation (ArrowUp/Down/Enter) works on the symptom grid.

  4b. BMI Calculator Widget:
    - Collapsible <details> widget on the symptoms page.
    - User enters weight (kg) and height (cm).
    - calculateBMI() computes: BMI = weight / (height/100)^2, rounded to 1dp.
    - Categories: Underweight (<18.5), Normal (18.5-24.9), Overweight (25-29.9),
      Obese (>=30).
    - Result displayed with color-coded label.
    - Validation: rejects zero/negative/NaN inputs with error message.

  4c. Severity Sliders:
    - When a symptom checkbox is checked, a range slider (1-3) is injected
      below it via injectSlider().
    - Slider labels: Mild / Moderate / Severe.
    - On form submit, JS reads each checked symptom's slider value and appends
      hidden inputs named severity_<symptom> with value "mild"/"moderate"/"severe".
    - On uncheck, removeSlider() removes the slider DOM element.

  4d. Form Submission:
    - POST /predict with selected symptoms + severity hidden inputs.

FEATURE 5 — DISEASE PREDICTION ENGINE
================================================================================

ROUTE:
  POST /predict → templates/result.html

EXECUTION FLOW:
  1. Reads selected_symptoms from form (list of symptom names).
  2. If empty → returns plain text error "Please select at least one symptom".
  3. Calls predict_disease(selected_symptoms, model, all_symptoms) from utils.py:
       a. create_input_vector(): builds a binary vector of length = total symptoms.
          Each position is 1 if that symptom was selected, else 0.
       b. Wraps vector in a pandas DataFrame with column names matching training data.
       c. model.predict() → top disease name.
       d. model.predict_proba() → probability array for all classes.
       e. Top 3 diseases extracted by sorting probabilities descending.
       f. Returns: (disease, confidence%, important_symptoms, top3_list).
  4. Reads severity data from hidden form inputs (severity_<symptom>).
  5. Looks up disease_info.json for description, causes, precautions, medicines.
  6. Generates a UUID share_token for the shareable link.
  7. Inserts prediction record into predictions table:
     (email, disease, confidence, symptoms_csv, severity_json, share_token).
  8. Renders result.html with all prediction data.

ML MODEL (model.py — training script):
  - Loads Training.csv, fills NaN with 0.
  - Trains 3 models: RandomForestClassifier, GaussianNB, GradientBoostingClassifier.
  - Evaluates each on 20% test split: accuracy, precision, recall, confusion matrix.
  - Selects best model by accuracy.
  - Saves best model to models/model.pkl via pickle.
  - At app startup, model.pkl is loaded into memory.

DATABASE:
  Table: predictions (id, email, disease, confidence, symptoms, created_at,
                       bookmarked, severity_data, share_token)

FEATURE 13 — AI CHATBOT (Dr. Atlas)
================================================================================

ROUTE:
  POST /chat → JSON {"reply": "..."}

EXECUTION FLOW:
  1. Checks session; returns error JSON if not logged in.
  2. Reads message and optional context from POST form data.
  3. Greeting detection: if message is a greeting (hi, hello, hey, etc.) →
     returns a fixed welcome reply without calling Ollama.
  4. Basic English word check: message must contain at least one word from a
     predefined common_words set. If not → returns "I didn't understand that."
  5. ML symptom detection from message text:
     - Scans message for symptom keywords matching all_symptoms list.
     - If symptoms found → calls predict_disease() to get top 3 predictions.
     - Formats as "ML Prediction based on detected symptoms: ..." text.
  6. Builds Ollama prompt:
     - Optional context prefix (last bot message, up to 500 chars).
     - Instructs model to list conditions, what to do, when to see a doctor.
  7. POST to Ollama API (http://localhost:11434/api/generate):
     - Model: tinyllama
     - num_predict: 250, temperature: 0.4, top_p: 0.9
     - Timeout: 60 seconds.
  8. Parses raw response: splits into sentences, divides into thirds:
     - Section 1: "🔍 Possible Conditions"
     - Section 2: "💊 What To Do"
     - Section 3: "🏥 See a Doctor If"
  9. Appends ML prediction text if symptoms were detected.
  10. Follow-up question: if symptoms detected in message, looks up FOLLOWUP_MAP
      for a relevant follow-up question and appends it to reply.
  11. Persists both user message and bot reply to chat_messages table.
  12. Error handling:
      - Timeout → "⏳ AI is taking too long."
      - Ollama not running → "⚠️ Ollama is not running. Start it with: ollama run tinyllama"

  Ollama Auto-Start:
    - On app startup, a daemon thread tries to launch ollama.exe from the
      local Programs path using subprocess.Popen.

FOLLOWUP_MAP:
  Maps symptom keywords (fever, headache, cough, chest_pain, fatigue, nausea,
  vomiting, dizziness, rash, shortness_of_breath, back_pain, stomach_pain,
  sore_throat) to specific follow-up questions.

FRONTEND (floating chatbot on symptoms page):
  - Floating action button (FAB) with 🤖 icon.
  - toggleChat() opens/closes the chat window.
  - toggleMaximize() expands chat to full-screen mode.
  - On open: fetches last 20 messages from /api/chat_history and prepends them
    to the chat box (chronological order).
  - sendMessage(): POSTs message + context to /chat, shows typing indicator,
    renders formatted reply.
  - formatReply(): parses reply lines into section titles, bullet points, and
    plain lines with appropriate CSS classes.
  - "📋 Select Symptoms for Prescription" link appended to every bot reply.
